# Interaction Library Scaffold ‚Äî Design

**Date:** 2026-07-06
**Status:** Approved
**Scope:** First scaffold pass ‚Äî infrastructure + one reference primitive (Button)

## Vision

A library of interaction primitives (button, select, dialog, ‚Ä¶) where each primitive is
engineered as a *system* ‚Äî interaction physics, motion, accessibility, and state
orchestration treated as first-class concerns ‚Äî not a component with variants. Distributed
via the shadcn registry so consumers install source, not a package.

This pass builds the machine that produces primitives: repo conventions, registry
pipeline, docs system, site shell, and one reference primitive proving the template
end-to-end. Deep-dive craftsmanship on each primitive comes later, one primitive at a
time.

```
‚îÇ       ‚îú‚îÄ‚îÄ use-button.ts      #   behavior hook

‚îú‚îÄ‚îÄ lib/                       # site-only: fs loader, demo map ‚Äî never installable
‚îú‚îÄ‚îÄ app/                       # routes (see Site shell)
‚îú‚îÄ‚îÄ registry.json              # shadcn manifest, multi-file items

```

The `registry/rsr/{lib,hooks,ui}` shape is deliberate: the shadcn CLI rewrites
imports only for paths matching `registry/<namespace>/<bucket>/‚Ä¶`, so in-repo imports
like `@/registry/rsr/hooks/use-press` are rewritten to the consumer's `@/hooks/use-press`
on install ‚Äî no tsconfig tricks needed on either side.

Rule: `registry/**` must never import from `app/` or `lib/` (site code). Site code may
import from `registry/**` freely (demos, docs rendering).

## Registry pipeline

- `registry.json` (schema `https://ui.shadcn.com/schema/registry.json`, name `@rsr`)
  declares each primitive as a multi-file item. Shared lib files are their own items
  (`motion-tokens`, `use-press`, `cn`) referenced via namespaced
  `registryDependencies` (`@rsr/use-press`), so installing `button` pulls its whole
  system through one registry mapping.
- `bun run registry:build` ‚Üí `shadcn build` ‚Üí `public/r/{name}.json`.
- Consumers add one mapping to their `components.json` ‚Äî
  `"registries": { "@rsr": "https://<site>/r/{name}.json" }` ‚Äî then
  `bunx shadcn@latest add @rsr/button`. Direct URLs also work.
- Build-time validation: every `registry.json` item's files must exist on disk; every
  `registry/{primitive}/` folder must have a manifest entry ‚Äî fail the build loudly on
  drift.

Landing (minimal this pass): wordmark, tagline ("Components are easy to copy.
Interaction isn't."), one small interactive moment ‚Äî the gradient accent in the wordmark
responds to pointer position (per the branding direction: the logo itself is
interactive) ‚Äî and a link into docs.

### Primitive doc page anatomy

Assembled by the loader; renders in order:

1. Title, one-line description, install command with copy button.
2. Live demos ‚Äî every `demos/*.tsx` in a preview frame with code toggle.
3. MDX body with custom components for: **Interaction spec** (press/hover/cancel/
   interrupt behavior), **Accessibility** (keyboard map, ARIA, SR behavior), **Motion**
   (what animates, why, reduced-motion), **API reference** (props table).
4. Source viewer ‚Äî all files in the system folder, tabbed.

### Loader (`lib/registry-loader.ts`)

Craft's `diary.ts` evolved to folder-aware: given a slug, reads all source files under
`registry/{slug}/`, demo files under `demos/`, and `content/primitives/{slug}.mdx`.
Graceful degradation: missing MDX ‚Üí render demos + source only; missing demos ‚Üí dev
warning, page still builds.

## Theming / branding

- Tailwind 4 `@theme` tokens in `globals.css`: monochrome neutral scale, **light-first**
  (per the owner's minimal references and the black-on-light logo), dark via
  `next-themes` toggle. (Supersedes the earlier dark-first decision.)
- One accent: subtle gradient defined as tokens; the logo's star carries the gradient
  splash. Accent used only for active states and focus rings.
- Typography: **Satoshi** (local OTFs in `public/font/satoshi/`, via `next/font/local`)
  as sans; Geist Mono for code/labels. Logo: `components/logo/logo.tsx`.
- UI stays minimal throughout: generous whitespace, soft muted surfaces, small
  uppercase mono labels, no heavy chrome.

## Reference Button (scaffold-level scope)

Built on native `<button>` + `use-press` (Base UI has no button primitive; Base UI
enters with Dialog/Select later).

- Variants + sizes (cva), styling reacts to `data-pressed` / `data-hovered` /
  focus-visible via data-attributes.
- Disabled semantics, `prefers-reduced-motion` respected via motion tokens.
- Explicitly out of scope this pass: async/loading states, success/error choreography ‚Äî
  that's the Button deep-dive, a later project.

### Shared lib starter

- `motion/tokens.ts` ‚Äî named durations, easings, springs; reduced-motion-aware helper.
  No magic numbers in components.
- `interaction/use-press.ts` ‚Äî pointer + keyboard press: starts on pointer-down, cancels
  when dragged off, Enter/Space parity, exposes pressed state as data-attribute.
- `utils/cn.ts`.

## Verification (definition of done)

1. `bun run build` passes clean.
2. `bun run registry:build` emits valid JSON to `public/r/`.
3. End-to-end: in a scratch project with `"registries": { "@rsr":
   "http://localhost:3000/r/{name}.json" }`, `bunx shadcn@latest add @rsr/button`
   installs Button **with** its lib dependencies and correctly rewritten imports.
4. `/primitives/button` renders: live interactive demo, MDX sections, source viewer.

No test framework this pass; vitest + testing-library arrive with the first primitive
deep-dive, where behavior contracts (press semantics, keyboard parity) get real tests.
